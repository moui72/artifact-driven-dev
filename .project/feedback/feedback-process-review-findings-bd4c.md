# Feedback

## Bugs
- [x] `/ardd-codify` (existing-project onboarding path) never generates `WORKFLOW.md`, unlike `/ardd-bootstrap` (greenfield path) — an asymmetry between the two supported entry points that neither guide calls out. Fix: add a WORKFLOW.md-generation step to `/ardd-codify`, reusing the same skills-table content as bootstrap's template.
- [x] `/ardd-analyze`'s auto-trigger-on-tasks-completion is implemented two different ways for the same event: `/ardd-converge` explicitly instructs "run /ardd-analyze now" when a tasks file flips to `completed`; `/ardd-implement` instead relies on the next loop iteration discovering all tasks are done, with no explicit instruction at the actual completion step. Fix: make `/ardd-implement` explicit at its own completion step, matching `/ardd-converge`'s phrasing.
- [x] `project-lock.sh` exists for same-directory concurrency but `/ardd-feature` and `/ardd-sync` — the two skills most likely to race on `features.md` — never call `check`/`touch`, unlike `/ardd-plan`/`/ardd-tasks`/`/ardd-implement`/`/ardd-converge`. Fix: add `project-lock.sh check ardd-feature` / `check ardd-sync` before their `features.md` read-modify-write, and `touch` after.
- [x] Nothing detects "this feature slug is already in flight on another branch/worktree" before `/ardd-plan` starts — two parallel worktrees can independently plan and implement the same slug, surfacing only as a merge conflict (or duplicate completed work) after the fact. Fix (highest-value): extend `branch-info.sh` to enumerate other worktrees via `git worktree list --porcelain` and expose their `features.md` slug statuses; have `/ardd-plan` step 3a's existing slug lookup check this too, printing an advisory (never blocking).
- [x] `lint-project.sh` doesn't catch two non-superseded plans (`draft`/`approved`) both listing the same feature slug in their `features:` frontmatter — a pure-file-based signal (no worktree logic needed) that two branches independently planned the same feature. Fix: add this as a rule alongside the existing "approved plan but feature still backlogged" check.
- [x] `/ardd-sync push`'s dedup logic (`sync-slug-match.sh`) is designed for crash-retry idempotency, not true concurrent-run safety — two genuinely simultaneous `push` runs can both search, both find no existing issue, and both create duplicate GitHub issues for the same slug. Fix: folds into the `project-lock.sh` fix above; document the residual external-system race as a known limitation distinct from the crash-retry case already handled.
- [x] Branch-check logic is prose-duplicated across `/ardd-plan`, `/ardd-tasks`, `/ardd-implement`, and the three copies have already silently diverged — `/ardd-plan`'s version has extra slug-derivation logic the other two lack, while `/ardd-tasks` and `/ardd-implement`'s copies are closer to verbatim duplicates of each other. CLAUDE.md documents the duplication itself as deliberate (skills can't call each other as subroutines), but the drift between the three copies isn't. Fix: add an explicit cross-reference comment in each of the three SKILL.md files noting they must stay behaviorally identical except for the documented slug-derivation addition, so an editor touching one is prompted to check the others.

## UX
- [x] Confusing near-duplicate skill names/roles: `/ardd-feature` (log an idea to backlog) vs `/ardd-featurize` (one-time backfill of features.md from an existing codebase) differ by only "-ize" but do unrelated things. Separately, four "check things" skills (`/ardd-analyze`, `/ardd-lint`, `/ardd-verify`, `/ardd-critique`) have no side-by-side comparison anywhere in the docs. Fix: add a comparison table to USAGE.md distinguishing all four, and a one-line disambiguation note in `/ardd-feature`'s and `/ardd-featurize`'s own SKILL.md pointing at each other.
- [x] `lint-project.sh` has zero coverage of `critique.md`, `DEFECTS.md`, `SYNC.md`, `STATUS.md` (each with its own informal date-stamp convention). Likely deliberate, but undocumented as such — a reader assumes it's an oversight. Fix: add a one-line comment in `lint-project.sh`'s header explaining the omission is intentional.
- [x] `/ardd-add-artifact` is far terser than `/ardd-bootstrap` for a structurally similar job — it never explicitly states which frontmatter fields must be set, where sibling skills like `/ardd-codify` are explicit about this. Fix: add an explicit frontmatter-fields-to-set step, matching `/ardd-codify` step 4's level of explicitness.
- [x] `/ardd-feature`'s name overpromises — it only logs a backlog entry with no artifact edits, but the name reads like it does the feature. Fix: consider a louder one-line callout at the top of its SKILL.md making the "logging only, no design work" scope explicit.
- [x] Feedback items use a 2-state checkbox (`[ ]`/`[x]`) while critique findings use a 3-state convention (adds `[-]` for rejected) — no "won't fix" state exists for feedback, training users on two different resolution vocabularies for similar concepts. Fix: consider extending feedback's checkbox convention to match critique's 3-state model, or document why the asymmetry is intentional.
- [x] `project-lock.sh` provides zero protection across `git worktree` checkouts (its lock file lives inside each worktree's own `.project/`), and nothing documents this limitation — multi-worktree users may believe they're protected when they aren't. Fix: add one sentence to `project-lock.sh`'s header and README's worktree guidance stating the blind spot explicitly.
- [x] No documented guidance exists for resolving git merge conflicts in single-writer report files (`STATUS.md`, `DEFECTS.md`, `SYNC.md`, `critique.md`) — users may try to hand-merge these instead of taking either side and re-running the owning skill. Fix: add a short "resolving .project/ merge conflicts" note (take either side for report files and re-run the owning skill; resolve `features.md` textually since it's append-oriented, then run `/ardd-lint`).
- [x] Frontmatter status-enum transition comments (e.g. `status: draft # draft -> approved -> superseded`) are repeated as inline YAML comments across multiple SKILL.md files, duplicating what `lint-project.sh`'s header already declares as schema-of-record. If an enum value changes, every inline copy needs a matching edit or silently goes stale. Fix: keep one inline example per file for readability, but have each comment explicitly say "(schema-of-record: scripts/lint-project.sh)".
- [x] `project-lock.sh check <skill-name>` is invoked with a different literal skill-name string in each caller, with no single documented list of valid caller names — a future skill adding lock support has to reverse-engineer the convention from existing callers. Fix: document the caller-name convention once in `project-lock.sh`'s own header.
