# Feedback

## Bugs
- [x] F001 The `ardd-constitution-data` suggestion catalog's two commonly-paired items — Test-First Development (a Core Principle: every test is "written and confirmed to fail before any implementation code is written") and Pre-commit Enforcement (a Quality Standard: a hook runs the full test suite, bypass prohibited except documented emergency) — silently contradict when both are adopted, which is the natural default combination. A TDD red commit *is* a deliberately failing test; a full-suite pre-commit hook rejects it, leaving only bad escapes (`--no-verify`, which the same standard prohibits outside an emergency; squashing test+impl into one commit, which abandons the red-then-green history the TDD flow just produced; or not committing the red state at all). Reported by an Atelier project (ARDD consumer, solo workflow) that hit this live and had to bypass the hook with `--no-verify` on every routine TDD red commit. Fix: add a "committing the red state" note to the Test-First catalog entry (mark the red test with the test framework's expected-failure modifier — e.g. Vitest `test.fails`/`it.fails`, pytest `@pytest.mark.xfail(strict=True)`, Jest `test.failing`, RSpec `pending` — so the commit passes the hook as an "expected fail" without a bypass; the paired implementation commit removes the marker, and a stale marker left after the feature lands fails the suite loudly, self-enforcing removal). Add a matching note to the Pre-commit Enforcement entry stating explicitly that TDD red states are handled by the expected-failure marker, not the emergency-bypass clause, so that clause isn't quietly eroded by routine test-first work. Where a framework has no native xfail marker (JUnit 5, Go `testing`), name the fallback (squash test+impl into one commit, or a sanctioned red-commit bypass) so the contradiction is resolved for every paradigm the catalog might seed, not just Vitest-shaped ones. Empirically verified against Vitest v4.1.10: a red test marked `it.fails` reports "1 expected fail" and the suite passes; a green test that still carries a stale `it.fails` marker fails the suite with "Expect test to fail." - [x] F002 `ardd-implement`'s TDD flow (the step that says "for test tasks under a TDD paradigm: write the test first, confirm it fails, then stop at the red state — mark the test task complete," followed by the per-task commit step) doesn't account for a full-suite pre-commit hook: as generated, the red-state commit predictably fails such a hook and pushes the agent toward `--no-verify`. Fix: when a full-suite pre-commit hook is present (Pre-commit Enforcement adopted), the red commit should apply the test framework's expected-failure marker and the paired implementation task's commit should remove it — same resolution as F001, applied at the flow-execution level rather than just the catalog-suggestion level.
