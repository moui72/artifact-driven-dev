# Plan: v0.10.2 fixes (phantom-completion recovery)

## Goal

Actually land the three skill-prose edits (T006, T007, T010) that a
prior run marked complete without applying, add a real deterministic
verification gap-closer to `ardd-state.sh`'s task/tasks-file completion
flow so a wrongly-flipped file can't reach `completed` unnoticed, and
tighten `/ardd-implement`'s own process so this failure mode is caught
before a commit ever claims work that wasn't done — then cut v0.10.2.

## Scope

**In scope:**
- The three concrete missing edits from `feedback-prerelease-full-sweep-v0-10-1-e5d8.md`
  F001:
  1. `skills/ardd-status/SKILL.md` step 1's banner-line logic: handle
     `ardd-update-check.sh`'s `dev-ahead` outcome distinctly from
     `behind` (never recommend `/ardd-update` when it would regress a
     dev-mode checkout that's ahead of the latest tag).
  2. `skills/ardd-init/SKILL.md` step 7: add the diff-verification
     instruction before trusting a `feat:`-titled commit's message as
     evidence a capability shipped.
  3. `skills/ardd-plan/SKILL.md` step 10: actually gate the "view in
     browser?" question on the constitution's `plan_preview` field.
- `ardd-state.sh`'s `tasks-flip <file> completed` (F002): refuse the
  flip unless every task line's checkbox is already `[x]`, mirroring
  the illegal-transition refusals the script already performs
  elsewhere.
- `skills/ardd-implement/SKILL.md` step 8/9 (F003): before running
  `task-check`, require an explicit self-check that the task's stated
  target file(s) actually show the described change in `git diff` —
  made a mandatory, visible step in the skill's own prose, not a new
  script (verifying "does this diff match what this specific task
  described" is judgment against free-text task descriptions, not a
  mechanizable pure function — Principle II only mechanizes what's
  actually a pure function of file state).
- A regression test for F002's new `tasks-flip` refusal (mechanizable,
  test-first per Principle V).
- A version bump and changelog-equivalent note for the eventual v0.10.2
  cut (out of this plan's phases — a manual dispatch step at the end,
  same as v0.10.1's).

**Out of scope:**
- Any change to `ardd-state.sh task-check` itself beyond what F003
  requires in `/ardd-implement`'s prose — `task-check`'s own mechanics
  (flipping one checkbox) are correct and don't need a script change;
  the gap is upstream, in when/whether the agent calls it.
- Re-auditing the rest of `v0.10.1`'s changes for similar
  phantom-completions — T001-T005, T008, T009, T011, T012 were all
  individually confirmed correctly landed (by direct grep/diff, in the
  triage that produced this feedback) before this plan was drafted; no
  further audit needed.
- The `stable-release` branch-protection `required_linear_history` fix
  (already resolved live via `gh api`, not a code/skill change this
  repo's own `.project/` tracks).

## Technical Approach

The three F001 edits are pure prose corrections to existing SKILL.md
files — the same category of change the original (broken) run was
supposed to make; this plan just makes them for real, with each task's
completion explicitly checked against a `git diff` of the target file
before `task-check` runs (dogfooding F003's own fix while implementing
it).

F002's fix is a straightforward addition to `ardd-state.sh`'s existing
`tasks-flip` case statement: before the `in-progress -> completed`
transition, grep the tasks file for any `- [ ]` line (unchecked) and
refuse with a clear message naming the still-open task ID(s) if any
remain — mirroring the script's existing refusal style (e.g.
`ready -> completed refused (skips in-progress)`).

F003 is a prose addition to `/ardd-implement`'s step 8: immediately
before calling `ardd-state.sh task-check`, the agent states which
file(s) the task named and shows (or explicitly reasons about) the
actual `git diff` for those files, confirming the change described in
the task is genuinely present — a lightweight, mandatory self-check
step, not a new mechanism.
