# Plan: uncommitted plan/tasks delegation gap + TDD/pre-commit xfail contradiction

## Goal

Close two feedback-reported gaps: (1) `/ardd-implement`'s delegation gate can
launch a worktree subagent against an uncommitted plan/tasks file and only
discover the failure after a full agent-launch round trip; (2) the
`ardd-constitution-data` catalog and `ardd-implement`'s TDD flow don't name
the expected-failure-marker (`xfail`) resolution for the routine
Test-First + full-suite-pre-commit-hook combination, leaving downstream
projects to discover the contradiction live and reach for `--no-verify`.

## Scope

**In scope:**
- `/ardd-implement`'s delegation gate: a pre-flight uncommitted-files check
  against the chosen tasks file and its bound plan, before launching a
  worktree subagent.
- `templates/constitution-suggestions.md`: add a "committing the red state"
  note to the Test-First Development entry, and a matching note to the
  Deterministic Gates entry (the catalog's actual pre-commit-hook entry —
  the feedback's "Pre-commit Enforcement" name is this repo's own
  constitution's paraphrase of it, not a separate catalog entry) stating TDD
  red states are handled by the marker, not the emergency-bypass clause. Add
  the framework-mapping table (Vitest/pytest/Jest/RSpec have a marker; JUnit
  5/Go `testing` don't — name the fallback).
- `.claude/skills/ardd-implement/SKILL.md` step 6's TDD bullet: when a
  full-suite pre-commit hook is present, apply the framework's
  expected-failure marker on the red commit and remove it on the paired
  green commit.

**Out of scope:**
- Changing this repo's own `.project/artifacts/constitution.md` — its
  existing Quality Standards paradigm (fixture-based regression tests,
  test added in the same commit as the check) doesn't produce a separately
  committed red state in the first place, so the contradiction described in
  the feedback doesn't apply to ArDD's own development workflow. Both
  feedback items are about what ArDD *seeds into or executes for* consumer
  projects, not about ArDD's own constitution — neither task touches
  `.project/artifacts/`.
- Auto-detecting whether a target project's pre-commit hook actually runs
  the full suite (out of scope — the catalog/skill text names the
  resolution; applying it in a given project is a human/agent judgment
  call already covered by "Check `constitution.md` ... before touching a
  test task").

## Technical Approach

**Delegation pre-flight check:** `/ardd-implement`'s delegation gate (step 3
preamble, where it currently calls `inflight-worktrees.sh` and decides
whether to delegate) gains a `git status --short <plan file> <tasks file>`
check against the chosen tasks file and its bound plan (`plan:` frontmatter)
before launching a worktree subagent. If either is untracked or modified
relative to HEAD, block delegation and tell the user to commit first (or,
on confirmation, commit them) rather than discovering the gap inside a
subagent after a full launch. This mirrors the existing
`worktree-align.sh`/`fold-to-main.sh` discipline of refusing rather than
silently working around a bad precondition.

**Catalog + skill xfail resolution:** the fix is textual — add the
"committing the red state" guidance where Test-First Development and
Deterministic Gates are defined (`templates/constitution-suggestions.md`),
and mirror the operative instruction in `ardd-implement`'s TDD execution
step, gated on "a full-suite pre-commit hook is present" (i.e. Deterministic
Gates was adopted) so the guidance doesn't fire for projects that never
took that quality standard.
