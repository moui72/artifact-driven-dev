# Plan: fix new.sh's Codex-launch stdin error

## Goal

Make `new.sh`'s kickoff `launch()` step work under `--harness codex` the
same way it already works under Claude Code, instead of failing with
`Error: stdin is not a terminal`.

## Scope

In scope: `new.sh`'s `launch()` function and its `exec` call; a static
regression guard mirroring `test-new.sh` case 14's existing discipline
(no live tty in CI, so the fix is pinned by grepping the source line, not
by executing it end-to-end); `docs/install.md`'s `new.sh` kickoff
documentation, if it describes the exec's stdin handling in a
harness-neutral way that would now be inaccurate.

Out of scope: any other part of `new.sh`'s Codex support; the `ask_kickoff`
prompt itself (already correctly reads `/dev/tty`, unaffected); a general
harness-abstraction refactor of `launch()` beyond this one branch.

## Technical Approach

Empirically confirmed (this session, local `codex-cli 0.144.6`): piping an
EOF'd/non-tty stdin to `codex` reproduces the exact reported error —
`echo "" | codex '...' </dev/null` → `Error: stdin is not a terminal`.
Codex's CLI has no equivalent of Claude Code's `process.stdin.isTTY &&
process.stdout.isTTY`-checked fallback to opening `/dev/tty` itself (the
behavior `launch()`'s existing comment documents in detail for the Claude
Code case) — it appears to require stdin to already be a real terminal
and errors immediately otherwise.

`launch()` currently issues one unconditional `exec "$handoff_tool"
"$handoff_cmd"` for both harnesses, deliberately never redirecting stdin
— correct for Claude Code (a read-only `< /dev/tty` redirect would pass
Claude's `isTTY` check but then be silently ignored for keystrokes; the
correct fix there was documented as "leave stdin alone", not redirect
it). Codex needs the opposite: since it has no `/dev/tty`-fallback logic
of its own, it needs stdin explicitly connected to a real terminal via
`<> /dev/tty` (read-write, the same construct `new.sh`'s own comment
already identifies as the one that "makes it exit outright" for Claude —
a *different* CLI's failure mode there, but the mechanically correct
redirect for a CLI that has no isTTY-based fallback of its own).

**Open question flagged below**: this sandbox has no controlling
terminal at all (`test -t 0` is false, `exec 3<>/dev/tty` fails with
`device not configured`), so the `<> /dev/tty` redirect cannot be
live-verified against Codex's actual prompt/TUI behavior from here — only
the *failure* was reproduced, not the *fix*. This mirrors the repo's own
documented tty-testing trap (`new.sh`'s existing case-14 comment, and
CLAUDE.md's `tty_ok()` note): no tty exists in this environment to
exercise either. The task below must be verified live, in a real
terminal, by whoever implements it — not merely by a passing regression
test.

Given Principle VI (Simplicity/YAGNI), the fix stays minimal: a harness
`case` branch inside `launch()` choosing the redirect, not a broader
launch-abstraction refactor. No Complexity Tracking entry needed — this
is the same shape `launch()` already has for other harness differences
(`print_handoff_command`'s existing `case "$harness"`), not a new pattern.
