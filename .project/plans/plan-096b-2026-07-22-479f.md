## Technical Approach

**F001 — `/ardd-update` harness compat guard.** Read the real
current text (`skills/ardd-update/SKILL.md:105-128`): the guard at line
111-119 checks `<source>/install.sh` for `--harness` support **only
when `HARNESS=codex`** — the `HARNESS=claude` path falls straight to
line 121-122's unconditional `<source>/install.sh --harness <harness>
<target>`, so a pre-`--harness` source (e.g. v1.0.4) still gets `--harness
claude` passed to an installer that doesn't recognize the flag,
reproducing the exact reported crash. Fix: check `--harness` support
once, for both harnesses, before the reinstall line. Codex's existing
stop-and-explain behavior is unchanged (a pre-harness installer genuinely
can't preserve a Codex install safely). Claude's case is the new branch:
when unsupported, this is *safe* — a pre-harness `install.sh` always
installed for Claude implicitly (harness didn't exist as a concept yet)
— so simply omit `--harness claude` from the invocation rather than
stopping, and note in the relay that the source predates
explicit-harness recording.

**F002 — `ardd-scripts` silent wrong-CWD write.** Every `.project/`
path in `ardd-state.sh`/`project-lock.sh`/`source-resolve.sh` is
CWD-relative by design (grounded: `project-lock.sh`'s own doc comment
states `project-dir` "defaults to `.`", same convention as
`lint-project.sh`) — this is correct for normal skill usage, where CWD
is always the target project root, and a `--project-root`-everywhere
redesign would be disproportionate machinery for what's fundamentally a
caller-discipline problem (Simplicity/YAGNI). `feature-create` already
has a partial guard (`ardd-state.sh:265`: `[ -d ".project" ] || die
...`) — but it only catches "no `.project/` at all," not "a *different*,
also-valid `.project/`" (exactly what happened: the ArDD source repo
itself dogfoods its own `.project/`, so the guard passed while writing
to the wrong one). The proportionate fix: every write-performing
`ardd-state.sh` subcommand (`feature-create`, `stamp`, `tasks-flip`,
`task-check`, `feature-flip`, `feedback-mark`, `feedback-planned`, `mint`)
prints the **resolved absolute path** of what it just wrote, as part of
its existing confirmation line (e.g. `feature-create: wrote
/full/path/.project/features/foo.md`, not just the current bare
`feature-create: ...`). This doesn't prevent a wrong-CWD invocation, but
it turns a previously silent wrong-target write into an immediately
visible one — a human or agent skimming the transcript catches the
mismatch at the moment it happens (Principle II: deterministic
mechanization, not a prose reminder to "double-check your CWD").

**F003 — `/ardd-init` reverse-engineer overclaiming.** Add explicit
guidance to the existing-codebase reverse-engineer step: before writing
a universal-coverage claim ("every X has Y"), enumerate each instance of
X individually and verify Y against it — don't generalize from a
representative sample. When full verification isn't performed (or
isn't practical), phrase the claim conservatively ("N of M observed
entities have Y") rather than an unverified "every."

**F004 — `/ardd-plan` unhonorable TDD-red-first phrasing.** Add
guidance to step 12's task-generation rules: when drafting a red-first
task under a TDD paradigm, check whether its precondition work already
landed via an earlier task in the *same plan* (a phase before it, or an
earlier task in the same phase) — if so, don't phrase it as "confirm it
fails first," since there's no red state left to confirm; phrase it as
directly implementing/extending the already-existing code instead. This
is a recurrence of a 2026-07-15 finding class per F004's own report —
the fix generalizes the guidance so it doesn't keep recurring per-sweep.

**F006 — `/ardd-status` ambiguous zero-count omission.** The Feature
Backlog format (`skills/ardd-status/SKILL.md:213-219`) already documents
that only `retired`/`rejected`/`subsumed` omit at zero — the four core
buckets (backlogged/planned/tasked/implemented) are implicitly meant to
always print, but the wording never says so explicitly, leaving room for
an agent to misapply the omit-rule to all seven buckets. Fix: add one
explicit sentence stating the core four always print, even at zero —
only the three terminal buckets omit.

**Constitution compliance**: none of these five changes are code paths
with a declared testing paradigm requirement, except F002 (a real script
change) — Principle V (Deterministic Checks Are Test-First) applies
there; F001/F003/F004/F006 are skill-prose edits, which this repo's own
convention (see `CLAUDE.md`) states are explicitly not something
`scripts/test-*.sh` covers (skill *behavior* isn't script-tested, only
deterministic mechanics are).
