# Plan: rewrite `/ardd-*` invocation syntax to `$ardd-*` for Codex installs

## Goal

Make every skill's `/ardd-<name>` self-reference read as `$ardd-<name>`
in a Codex-harness install, so Codex sessions no longer see Claude Code's
invocation syntax quoted back at them.

## Scope

In scope: extending `install.sh`'s existing `install_skill_file()`
mechanism (which already rewrites `.claude/skills` path prose per
harness) with a second, narrowly-scoped rewrite pass for the codex
branch only — converting genuine `/ardd-<skill>` invocation references
to `$ardd-<skill>`, while leaving path references
(`.claude/skills/ardd-plan/SKILL.md` → already correctly rewritten to
`.agents/skills/...` by the existing pass) and non-invocation `ardd-*`
names (script filenames like `ardd-state.sh`, `ardd-update-check.sh`,
`ardd-version.md`, the `ardd-scripts`/`ardd-artifact-templates`
directories) untouched. A new regression test pinning both the
rewritten and untouched cases.

Out of scope: `new.sh`'s own printed-output `handoff_cmd` logic (already
harness-correct, unrelated to `SKILL.md` body content); README/USAGE/docs
site prose (source-side, human-facing, not installed — a separate,
not-yet-scoped concern); any change to which skill names exist or what
they do.

## Technical Approach

**Grounded in a direct grep of the actual pattern space** (this session):
across all 14 `skills/*/SKILL.md` files, `/ardd-*` occurs 327 times as a
genuine slash-invocation and 88 times as part of a `.claude/skills/`
path (the latter already handled correctly by the existing rewrite).
Critically, `/ardd-*` also names non-invocation entities that must
**never** be rewritten: `ardd-state` (a script, `ardd-state.sh`),
`ardd-update-check` (a script, and a literal substring of the real skill
name `ardd-update` — a naive prefix match would corrupt it),
`ardd-version` (`ardd-version.md`, a file), `ardd-scripts` and
`ardd-artifact-templates` (reference directories, not skills). Verified
empirically that none of these five non-invocation names currently
appear in a "looks like an invocation" context (backtick/space/paren
immediately before the slash) — so the fix is safe with an **explicit
allowlist of the 14 real skill names**, not a blanket `ardd-[a-z-]+`
wildcard, as a second line of defense beyond the boundary check alone.

This belongs in `install_skill_file()` in `install.sh` — the exact
function that already does this class of mechanical, harness-conditional
rewrite for path prose (Principle II: deterministic mutation over prose,
wherever the operation is actually mechanizable). Add a codex-only `sed`
pass, applied after the existing `.claude/skills` → `$SKILLS_REL`
substitution, matching `/ardd-<name>` only when:
- immediately preceded by a command-context boundary (backtick, space,
  `(`, `"`, or start-of-line) — this alone excludes every path-embedded
  occurrence, since a path always has a directory-name character
  immediately before that slash (e.g. the `s` in `.../skills/ardd-plan`,
  the `s` in `.../ardd-scripts/ardd-state.sh`);
- `<name>` is exactly one of the 14 real skill names (explicit
  alternation, not a wildcard);
- immediately followed by a non-identifier character (not a lowercase
  letter or hyphen) — this stops `/ardd-update` from matching inside
  `/ardd-update-check`.

Claude installs are untouched — the codex-only branch means Claude's
installed output stays byte-identical to today, a hard regression
requirement.

**Complexity Tracking justification** (Simplicity/YAGNI, Principle VI):
a boundary-plus-allowlist regex is more machinery than a plain
find/replace, but the alternative — hand-editing 327 occurrences across
14 prose files, with no mechanism stopping future drift as new
occurrences get added — is strictly worse complexity by the constitution's
own "deterministic mutation over prose wherever mechanizable" principle
(II). The regex's precision requirement (allowlist + boundary, not a
wildcard) is itself dictated by the five real false-positive risks found
during grounding, not speculative caution.

## Open Questions

None — the pattern space was grounded empirically this session (grep
counts, false-positive risk check) before drafting, per Principle II.
Also verified: the allowlist regex requires the literal substring
`ardd-` immediately after the slash, so it structurally cannot match any
built-in harness command (Codex's `/exit`, Claude Code's `/init`, etc.)
regardless of harness — confirmed by grepping every non-`ardd-`
`/<word>` occurrence across all skill files (12 found, all either URL
path fragments or prose words, none a real command reference). The one
near-miss, `ardd-init/SKILL.md`'s explicit mention of "Claude Code's own
built-in `/init` command, which is unrelated," stays accurate verbatim
in a Codex-installed copy — it already names Claude Code specifically,
not a generic "the built-in command."
