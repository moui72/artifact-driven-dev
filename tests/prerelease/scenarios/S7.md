# S7 — peripheral skills sweep

Axes: state = established real consumer (fresh clone); skills =
status/lint/defects/audit/feedback/refine/diagram/tracker. Smoke-tier.

## Purpose

Day-2+ skills on a real, lived-in `.project/`, judged on OUTPUT QUALITY —
are the reports incisive and current, or boilerplate and stale — not just
"did it error."

## Setup

- Clone a real ArDD consumer (default: `~/dev/atelier`) from its
  FILESYSTEM PATH into `$SCRATCH/s7/app`; `git remote remove origin`.
- Bring the install current: run the local source `install.sh` against it
  (upgrade quality is S3's job, not yours).

## Steps  (run in order; append to report after each)

1. `/ardd-status` — counts vs disk; is the summary something a returning
   maintainer could act on?
2. `/ardd-lint` — should pass or fail with messages that name file, line,
   field, and the allowed values (error-message quality was a 2026-07-15
   finding class).
3. `/ardd-defects` — do drift claims check out against the code? Verify
   two claims by hand. Then make a small innocuous code edit and rerun:
   is there any staleness signal problem (same-day staleness was a
   finding)? Revert the edit.
4. `/ardd-audit` — are findings specific to this project or generic
   checklist filler?
5. `/ardd-feedback` — capture one deliberately-planted observation; check
   the file's schema, `[artifacts: ...]` tags, and that lint accepts it.
6. `/ardd-refine` — a small real refinement to a non-constitution
   artifact; verify status/versioning handling.
7. `/ardd-diagram` — generate from an artifact with a `diagram_type`;
   is the Mermaid valid and does upsert hit the right destination
   without clobbering neighbors?
8. `/ardd-tracker` — NO real GitHub writes (guardrail 5). Exercise only
   what runs without a remote; expect a clean, comprehensible refusal or
   read-only report, not a crash.
9. Epic-label coverage: seed a couple of `epic:` fields into the
   consumer's feature register via `ardd-state.sh feature-field <file>
   epic <slug>` on two or three entries sharing the same epic value.
   Re-run `/ardd-status` and confirm its Feature Backlog section groups
   the backlogged/planned/tasked counts "by epic" correctly — one line
   per epic value seen, `implemented`/`retired` entries excluded from
   the grouping, and the plain (non-epic) counts still correct
   alongside it. Then run `/ardd-backlog --assign-epics` and confirm it
   behaves sanely against the already-`epic`'d entries — proposing
   nothing for slugs that already carry an `epic` value (or handling
   them per whatever its own documented behavior is; note which case
   applied).

## Scripted answers

- Any confirmation in any skill: accept the least-destructive option.
- Tracker push/sync offers: decline.
- Any next-step prompt: decline.

## Checklist

- [ ] each of the 8 ran to a coherent end (or refused clearly)
- [ ] defects claims verified; staleness behavior noted
- [ ] audit findings project-specific
- [ ] diagram upsert surgical
- [ ] single-writer ownership respected throughout (no skill wrote a
      report file it doesn't own)
- [ ] `/ardd-status` groups Feature Backlog counts "by epic" correctly
      after seeding `epic:` fields
- [ ] `/ardd-backlog --assign-epics` behaves sanely against
      already-labeled entries (proposes nothing, or handles them per its
      documented behavior — note which)
