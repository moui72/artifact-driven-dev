# S1 ā€” new.sh quickstart, greenfield

Axes: acquisition = `new.sh` (real network, `--beta`); state = greenfield;
mode = solo; execution = n/a (init only).

## Purpose

Exercise the real `curl | sh` acquisition path against the most recently
published beta, then `/ardd-init`'s greenfield interview, judging both the
acquisition UX and the quality of the seeded artifacts.

## Setup

- `$SCRATCH/s1/` empty. No git init ā€” `new.sh` owns project creation.

## Steps

1. Fetch and run the published quickstart for real:
   `curl -fsSL <raw stable-or-beta URL for new.sh> | sh -s -- --beta --no-kickoff $SCRATCH/s1/hello`
   (use `--no-kickoff`: you cannot answer a `/dev/tty` prompt). Record
   every line of output verbatim in your report; judge clarity.
2. Verify the result: `.claude/skills/ardd-*` present, `.project/ardd-version.md`
   records a beta tag, `.worktreeinclude` contains `.claude/skills/ardd-*/`,
   the printed `.gitignore` suggestion is exactly `.claude/skills/ardd-*/`
   (never broader ā€” that would be a release blocker).
3. In the new project, run `/ardd-init` for a trivial concept: a CLI that
   prints "Hello, <name>!" with a `--shout` flag. Answer its interview
   from the Scripted answers below.
4. Judge the seeded artifacts: is `constitution.md` proportionate to a
   toy CLI (over-built = a `ux` finding), statuses sensible, register
   empty or sensible?
5. Run `/ardd-lint` and `/ardd-status`; both should be clean/coherent on
   a fresh init.
6. Dynamic version badge, opt-in default: confirm step 1's install output
   (env unset) mentioned the `ARDD_VERSION_BADGE=1` opt-in ā€” a pointer
   line, not a printed snippet. Missing mention = a `ux` finding.
7. In the project, add a fake GitHub remote (never push anywhere in this
   scenario): `git remote add origin
   https://github.com/example-owner/example-repo.git`. Re-run the same
   install.sh the quickstart resolved (the checkout at `~/.ardd/source`)
   with `ARDD_VERSION_BADGE=1` set. The printed split-badge snippet must
   carry the real coordinates ā€” `example-owner/example-repo` and the
   project's current branch ā€” with no literal `OWNER`, `REPO`, or
   `BRANCH` placeholder anywhere in it.
8. Paste the printed snippet (including its
   `<!-- ardd-badge-version-start -->` / `-end` markers) into the
   project's README, then re-run the same `ARDD_VERSION_BADGE=1`
   install. With the markers present, the snippet must NOT be printed
   again ā€” a reprint is a finding.
9. Inside the README markers, hand-replace the version badge with a wrong
   one pointing at
   `img.shields.io/github/v/release/...artifact-driven-dev` (the source
   repo, not the project), and re-run the `ARDD_VERSION_BADGE=1`
   install. The output must include the advisory flagging the misdirected
   badge. No advisory = a finding.

(Steps 6ā€“9 are the smoke-budget slice of the badge surface. The full
badge-presence matrix ā€” no-README, previous-format pair/static badges,
bare misdirected badge, `/ardd-update` over a badged install ā€” lives in
S9, a targeted-only scenario run whenever the badge file-set changes.)

## Scripted answers

- new.sh kickoff prompt: pre-answered via `--no-kickoff`.
- `/ardd-init` interview: greenfield; concept as in step 3; language:
  POSIX sh or Python (your pick, say which); no extra artifacts beyond
  constitution; `workflow_mode: solo`; `next_step_prompt: false`;
  `delegation: ask`; `merge_policy: ask`.
- Any next-step / analyze handoff prompt: decline / stop.
