# S9 — badge matrix: install/update across badge-presence states

Axes: acquisition = local source `install.sh` (no network, fake remote);
state = matrix of README/badge states; mode = n/a (no `.project/`
lifecycle); execution = install + `/ardd-update`. **Targeted-only** — in
no smoke or full tier; run via `/scenario-sweep S9` whenever the badge
file-set changes (see the trigger note below).

## Purpose

Exercise the badge flow end to end against repos in every badge-presence
state — missing, current-format, previous-format, hand-rolled-wrong —
and judge that each install/update leaves the README well-formed and the
badge renderable. The 17 deterministic cases in
`scripts/test-install-version-badge.sh` already pin the mechanical
behavior; do NOT re-verify what a shell assertion can check in
isolation. Your job is the cross-state, end-to-end judgment: does the
whole flow (output text, snippet, advisory, resulting README) leave a
real consumer with a badge that renders, in every starting state?

**Trigger note (change-triggered scenario).** Rerun this scenario any
time the badge file-set changes: install.sh's `ARDD_VERSION_BADGE`
section, `templates/ardd-badge-workflow.yml`, `templates/ardd-badge.json`,
`templates/ardd-icon.svg`, or `templates/badge.md`. It never joins a
tier — its cost is only justified when that surface moved.

## Setup

- Local ArDD source checkout path supplied by the dispatcher (`$SRC`);
  all installs run `$SRC/install.sh` directly. Real network only for
  `img.shields.io`-shape inspection — never fetch, never push.
- One scratch repo per case, `$SCRATCH/s9/case<N>/`: `git init`, one
  commit, and a fake remote
  `git remote add origin https://github.com/example-owner/example-repo.git`.
  Note the branch name git chose — it must appear in badge URLs later.

After every install in a badged case, run the **well-formedness pass**
and record results in the report:

- Marker balance: for each of `ardd-badge-start`,
  `ardd-badge-version-start`, `ardd-badge-pair-start` present in
  README.md, exactly one matching `-end`, no nesting or duplication
  (e.g. `grep -c 'ardd-badge-version-start\|ardd-badge-version-end' README.md`
  and eyeball order).
- No placeholder residue: `grep -nE 'OWNER|REPO|BRANCH' README.md`
  must hit nothing.
- Badge lines are valid markdown image links —
  `[![alt](https://img.shields.io/...)](https://github.com/...)` — and
  the endpoint URL points at
  `raw.githubusercontent.com/example-owner/example-repo/<branch>/.github/badges/ardd-version.json`.
- The seeded JSON parses and carries the render-critical fields:
  `python3 -c 'import json;d=json.load(open(".github/badges/ardd-version.json"));print(sorted(d));assert d["labelColor"] and d["logoSvg"] and d["message"]'`.
- The workflow YAML parses (`python3 -c 'import yaml,sys;yaml.safe_load(open(".github/workflows/ardd-badge.yml"))'`
  if PyYAML is available; otherwise note that and judge by eye).
- The markdown around the insertion point is intact — no swallowed
  heading, no orphaned blank-line collapse, no badge line glued onto
  prose.

## Steps  (run in order; append to report after each case)

1. **Case 1 — fresh repo, no README.** `ARDD_VERSION_BADGE=1
   $SRC/install.sh` the repo. Expect: no `.github/badges/` or workflow
   writes, and a clear pointer that a README is needed first. Silent
   no-op or badge files written into a README-less repo = a finding.
2. **Case 2 — README, no badge at all.** Seed a small README (title,
   a paragraph, an `## Install` section). First run install with the
   env UNSET: expect the static-snippet suggestion plus the
   `ARDD_VERSION_BADGE=1` opt-in pointer, and no README edit ever
   (suggestion-only is the contract). Then re-run with
   `ARDD_VERSION_BADGE=1`: workflow + icon + JSON written; printed
   split-badge snippet carries `example-owner/example-repo` and the
   real branch, no literal placeholders. Paste the snippet (with its
   markers) under the title and run the well-formedness pass.
   Variant: in at least one badged case, set the remote to the
   scp-style ssh-config alias form
   `github-ardd:example-owner/example-repo.git` instead of the https
   URL and assert the coordinates still fill (no placeholder residue) —
   the parser reads owner/repo from the path, host token irrelevant.
3. **Case 3 — current-format badge already present** (continue in the
   case-2 repo). Re-run the `ARDD_VERSION_BADGE=1` install. Expect: no
   snippet reprint (markers present), every supporting file reported
   `already exists, left untouched`, README byte-identical
   (`shasum` before/after). Also re-run with the env UNSET over this
   badged repo and judge the output: does it wrongly re-suggest the
   static badge to a repo that already carries the version badge
   (the static check greps `ardd-badge-start`, which is NOT a substring
   of `ardd-badge-version-start`)? Note what happens either way.
4. **Case 4 — previous-format badges.** Two sub-repos: (a) the
   two-badge PAIR shape from `templates/badge.md`
   (`ardd-badge-pair-start` markers, coordinates hand-filled); (b) the
   stale STATIC-only badge (`ardd-badge-start` markers). Run
   `ARDD_VERSION_BADGE=1` install in each. Hard requirements: the
   existing badge block is never clobbered or edited, and nothing gets
   inserted into the README by the installer. Judgment call to record:
   in (a) the installer's marker check only knows
   `ardd-badge-version-start`, so it may print the split snippet at a
   repo already carrying the pair — is the output clear enough that a
   consumer wouldn't paste a duplicate? Confusing or
   duplication-inviting output = a `ux` finding.
5. **Case 5 — homespun wrong badge** (the real assisted-review failure
   mode). Two sub-repos: (a) a shields
   `github/v/release/...artifact-driven-dev` badge wrapped in
   `ardd-badge-version-start`/`-end` markers; (b) the same wrong badge
   bare, no markers. Run install in each — once env unset, once
   `ARDD_VERSION_BADGE=1`. The misdirected-badge advisory must fire on
   every run in both sub-repos; the wrong badge must be neither
   silently clobbered nor joined by a duplicate. Judge the advisory's
   remedy as a consumer would: is it self-sufficient (names the
   markers, the endpoint form, the JSON path, how to get the file), or
   does it require the ArDD source repo to decode?
6. **Case 6 — update path over a badged install.** In the case-2 repo
   (current-format badge, install recorded), run `/ardd-update`. Its
   `Source-Path` names a live checkout, so expect the dev-mode warning;
   answer proceed. After the update: badge files untouched or
   idempotently rewritten (diff them), no snippet reprint, no advisory
   (the badge is correct), README byte-identical. Run the
   well-formedness pass one last time.

## Scripted answers

- `/ardd-update` dev-mode warning: proceed.
- Any other confirmation: accept the least-destructive option.
- Any next-step / analyze handoff prompt: decline / stop.

## Checklist

- [ ] no-README repo: no badge files written, pointer message clear
- [ ] no-badge README: suggestion-only when env unset; opt-in pointer present
- [ ] snippet carries real owner/repo/branch, zero placeholder residue
- [ ] re-install over current badge: no reprint, files untouched, README
      byte-identical; env-unset rerun behavior noted
- [ ] pair-shape and static-only repos: never clobbered, never
      auto-edited; duplication-inviting output judged
- [ ] misdirected badge draws the advisory in both marker and bare
      placements, on both env settings; remedy self-sufficient
- [ ] `/ardd-update` over a badged repo: badge state stable, no advisory
- [ ] well-formedness pass green after every badged install: markers
      balanced, JSON parses with labelColor/logoSvg, workflow YAML valid,
      URLs shaped right, surrounding markdown intact
