# S8 ‚Äî Agent-tool worktree fan-out delegation

Axes: state = greenfield; mode = solo; execution = **real `Agent` tool,
`isolation: "worktree"`**, two `ready` tasks files backgrounded together
via `/ardd-implement`'s multi-select fan-out. Full tier.

## Purpose

S6 answers whether ArDD's scripts behave correctly against a manually
created second worktree; this scenario answers the harness-integration
question S6 deliberately punts on ‚Äî does `/ardd-implement`'s real
multi-select fan-out, dispatching genuine `Agent` (`isolation:
"worktree"`) subagents in parallel, land two independent runs' code +
state atomically on report-back, reap both worktrees post-merge, and
leave `inflight-worktrees.sh` empty afterward? It also folds in the
`fold-to-main.sh` eager-background-while-on-a-branch path (Decision
record 0004) as its setup step, rather than carrying a separate
scenario for it ‚Äî starting the second of the two runs from a feature
branch means backgrounding it exercises the fold path too.

## Setup

- `git init $SCRATCH/s8/app` (no remote). Install ArDD from local
  source.
- `/ardd-init`: trivial concept (e.g. a unit-conversion CLI), solo,
  `next_step_prompt: false`; `delegation: eager`; `merge_policy: auto`.
- `/ardd-backlog` two small, independent features (different files, no
  shared dependency ‚Äî the plan's parallel-safety rule). `/ardd-plan`
  each, approve both, so two separate `ready` tasks files exist, each
  bound to its own plan/feature slug. Also create one featureless
  chore-style plan + tasks file (`features: []`, e.g. "add a README
  usage section") ‚Äî it is never delegated; it exists so the Work Queue
  verdict checks below can observe the `features: []` ‚Üí `none` case.
- Before backgrounding the second run, `git checkout -b
  <second-run-semantic-branch>` on the primary checkout (so the second
  run starts from a feature branch, not the default branch) ‚Äî this is
  the setup step that exercises `fold-to-main.sh`'s eager-background-
  while-on-a-branch path when that run is backgrounded.

## Steps

1. Run `.claude/skills/ardd-scripts/inflight-worktrees.sh` first ‚Äî with
   nothing yet delegated it should report empty. Confirm two `ready`
   tasks files exist via `/ardd-implement --list` (three including the
   undelegated chore file).
2. Work Queue / parallel-matrix, pre-flight: run `/ardd-status` and
   verify the Work Queue section shows pairwise verdicts for the ready
   tasks files ‚Äî `independent` for the disjoint two-feature pair, and
   the featureless chore plan's entry reading `none` for its features
   (never `unknown`; `unknown` here is a finding).
3. Invoke `/ardd-implement`. It should offer multi-select delegation
   since more than one `ready` tasks file exists ‚Äî verify the fan-out
   multi-select picker displays the matrix annotations (each candidate
   line carries its pairwise verdict) ‚Äî select the two feature-bound
   files, leaving the chore file undelegated.
4. For the second run specifically (started while on the feature branch
   created in Setup): confirm the delegation gate runs
   `fold-to-main.sh` before dispatching it ‚Äî capture its `folded=true`
   output verbatim, and confirm the focused session returns to the
   local default branch before the worktree subagent is created.
5. Dispatch both as real `Agent` tool calls with `isolation: "worktree"`
   in parallel (a single message, two tool uses ‚Äî solo mode's delegation
   gate should offer this eagerly for both, regardless of which run was
   on a branch). Each subagent must run `worktree-align.sh` as its
   mandatory first act; require `aligned=true` for both before either
   proceeds to task work.
6. Work Queue, in flight: while both runs are still in flight (before
   either report-back), re-run `/ardd-status` and verify each delegated
   tasks file ‚Äî still `ready` on this checkout but claimed by an
   in-flight worktree ‚Äî reads `claimed`, reported as "claimed by
   in-flight worktree", in the Work Queue verdicts; the undelegated
   chore file must not read `claimed`.
7. Let both subagents complete their (small) tasks for real ‚Äî implement,
   check the checkbox, flip `‚Üícompleted`, run
   `sibling-tasks-complete.sh`, flip the bound feature
   `tasked‚Üíimplemented`, commit on the worktree branch ‚Äî per
   `/ardd-implement`'s delegated flow.
8. As each subagent reports back, play the coordinator: check
   `core.bare` is not `true` on the primary checkout, merge per
   `merge_policy`, and run `worktree-reap.sh` after each successful
   merge. Confirm both merges land code + state atomically (one merge
   per run, both present) and that the two report-backs ‚Äî arriving
   independently ‚Äî each serialize their own merge without clobbering
   the other's `STATUS.md`/register write (the merge driver's job, if
   configured; otherwise the take-either-side disposable-report rule).
9. Post-merge checks: `inflight-worktrees.sh` reports empty;
   `worktree-reap.sh --dry-run` lists no candidates; both worktrees are
   gone from `git worktree list`; both branches deleted via
   `git branch -d`; `completion-flip-check.sh` is silent against both
   completed tasks files.

## Scripted answers

- Delegation gate: background both, eagerly.
- Any merge_policy confirmation: accept the auto/eager path (least
  destructive).
- Any next-step prompt: decline.
