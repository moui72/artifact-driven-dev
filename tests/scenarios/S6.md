# S6 — delegation mechanics, script layer

Axes: state = greenfield; mode = solo; execution = **second worktree,
created manually** (`git worktree add`), exercising every deterministic
script in the delegation/merge/reap chain against a real second worktree.

## Scope note (why this is NOT an `Agent`-tool test)

This scenario deliberately does **not** use the `Agent` tool's
`isolation: "worktree"` mechanism. Two runs (2026-07-15, 2026-07-17)
established that no sweep-dispatch arrangement can exercise that
mechanism against the scratch project: a background subagent puts the
worktree call 2 levels deep (where a silent collapse onto the primary
checkout was observed once, root cause unresolved), and a top-level
dispatch's `isolation: "worktree"` binds to the ArDD repo itself — there
is no parameter to target a foreign repo. The open harness question
lives in the README's Coverage backlog as a manual-interactive recipe;
it is explicitly out of scope for automated sweeps. What IS testable —
and what this scenario answers — is: do ArDD's own scripts
(`worktree-align.sh`, `sibling-tasks-complete.sh`, merge,
`worktree-reap.sh`, `inflight-worktrees.sh`, `completion-flip-check.sh`)
behave correctly against a genuine second worktree, and does one merge
land code + state atomically?

## Setup

- `git init $SCRATCH/s6/app` (no remote). Install ArDD from local source.
- `/ardd-init`: trivial concept (e.g. temperature converter CLI), solo,
  `next_step_prompt: false`; `delegation: ask`; `merge_policy: ask`.
  `/ardd-backlog` one feature; `/ardd-plan`,
  approve. Should yield one `ready` tasks file with one small task.

## Steps

1. From the primary scratch checkout: `git worktree add ../app-wt -b
   <semantic-branch> <default>` — a real second worktree. Confirm via
   `git worktree list`.
2. Inside the worktree, run `worktree-align.sh` as the mandatory first
   act — use the primary checkout's absolute-path copy under
   `.claude/skills/ardd-scripts/` (a plain `git worktree add` does not
   carry gitignored ardd files in; this exercises the documented
   absolute-path fallback for real). Capture output verbatim; require
   `aligned=true`. Also verify the positive
   "am I actually in a separate worktree" check passes (it should — this
   IS a separate worktree; a failure here is a real ArDD bug).
3. Play the delegated subagent, entirely inside the worktree, per
   `/ardd-implement`'s delegated flow: flip tasks `ready→in-progress`,
   implement the task for real (verify the code actually works), check
   the checkbox, flip `→completed`, run `sibling-tasks-complete.sh`,
   flip the feature `tasked→implemented` via `ardd-state.sh`. Commit
   everything on the worktree branch.
4. Play the coordinator, back in the primary checkout: check
   `core.bare` (`git config --get core.bare` — must not be `true`);
   merge the branch (capture whether it fast-forwarded and that code +
   state landed together in ONE merge); run `worktree-reap.sh` (capture
   output verbatim; require `reaped=true`, branch deleted via
   `git branch -d`).
5. Post-merge checks: `inflight-worktrees.sh` reports nothing in
   flight; `completion-flip-check.sh` is silent against the completed
   tasks file (no orphaned flip — it already rode the merge).
